# Phase 4: Report and Landing - Context

**Gathered:** 2026-03-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Two deliverables: (1) `honeyprompt report` CLI subcommand that queries the SQLite event store and generates a structured Markdown disclosure artifact with executive summary and evidence table, and (2) the honeyprompt.sh landing page generated by the tool itself (dogfooding) with embedded canaries that prove HoneyPrompt works on itself.

</domain>

<decisions>
## Implementation Decisions

### Report structure
- **D-01:** Executive summary section at the top: date range, total detections (session-based), unique sessions, per-tier breakdown, known-crawler vs unknown agent counts.
- **D-02:** Evidence table below the summary: one row per detected session showing tier, proof level, timestamp, source IP, full user-agent, session ID, fire count, classification.
- **D-03:** Full metadata in report — no anonymization. IP, UA, and headers included as-is. The evidence must link callbacks to specific entities to be useful; anonymizing defeats the purpose.
- **D-04:** Session-based counts (not raw callback rows) per Phase 2 D-08. Known-crawler sessions labeled but presented separately from agent detection sessions.

### Report output
- **D-05:** `honeyprompt report` writes a Markdown file by default. Output file name is configurable via `--output <path>` flag (default: `report.md` in the project directory).
- **D-06:** `--stdout` flag prints the report to stdout instead of writing a file. Enables piping to other tools.

### Landing page
- **D-07:** honeyprompt.sh landing page is generated by running `honeyprompt generate` with a honeyprompt-specific config. The tool eats its own dogfood — the landing page IS a honeypot page with project branding.
- **D-08:** Landing page contains the same embedded canaries as any generated honeypot, proving the tool works on itself.

### Claude's Discretion
- Report Markdown formatting details (headings, table alignment, frontmatter)
- Default output filename and path conventions
- Landing page config specifics (which tiers/payloads to include, callback URL)
- Report CLI args struct details beyond --output and --stdout
- Whether to include a "generated by HoneyPrompt" footer in the report

### Deferred
- **Per-payload breakdown** — Report section organized by payload/embedding location showing which agents triggered each. Deferred to v2.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `src/store/mod.rs`: `count_detections()` already computes session-based detection counts excluding known crawlers. Report queries extend this pattern.
- `src/types.rs`: AppEvent struct has all fields needed for report rows
- `src/generator/mod.rs`: Full generation pipeline reusable for landing page creation
- `src/config/mod.rs`: Config struct with `bind_address`, `port`, `callback_url` — landing page config follows same pattern
- `src/cli/mod.rs`: Clap `#[derive(Parser)]` pattern for adding ReportArgs

### Established Patterns
- Clap subcommands with args structs (ServeArgs, MonitorArgs → ReportArgs)
- rusqlite parameterized queries via `params![]`
- rust-embed for binary-embedded assets
- TOML for configuration

### Integration Points
- `src/main.rs` needs `Report(ReportArgs)` subcommand dispatch
- `src/store/mod.rs` needs new query functions for report data extraction (sessions, events by session, metadata)
- `src/lib.rs` needs `pub mod report` module

</code_context>
