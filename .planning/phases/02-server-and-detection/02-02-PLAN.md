       - Uses INSERT ... ON CONFLICT(nonce) DO UPDATE pattern (upsert):
         ```sql
         INSERT INTO events (nonce, tier, payload_id, embedding_loc, first_seen_at, last_seen_at, fire_count, is_replay, session_id, remote_addr, user_agent, extra_headers)

           last_seen_at = ?5,
           fire_count = fire_count + 1,
           is_replay = 1

Since the schema does NOT have a classification column, store it in the extra_headers JSON blob as `{"classification": "...", "headers": {...}}`. This avoids schema migration.

       Actually, simpler approach: the `extra_headers` column already exists as TEXT. Serialize a JSON object containing both headers and classification:
       ```rust

           "classification": classification,
           "headers": headers_map

    Create `src/broker/mod.rs` implementing the D-01 broadcast architecture:

    1. Define `pub async fn broker_task(mut callback_rx: tokio::sync::mpsc::Receiver<RawCallbackEvent>, event_tx: tokio::sync::broadcast::Sender<AppEvent>)`:
       - Loop: `while let Some(raw) = callback_rx.recv().await`
       - Compute session_id via `crate::fingerprint::compute_session_id(&raw.fingerprint.source_ip.to_string(), &raw.fingerprint.user_agent)`
       - Create AppEvent from RawCallbackEvent + session_id, with is_replay=false and fire_count=1 (actual values come from DB writer response, but for broadcast we send the initial event; DB writer updates the truth-of-record)
       - `let _ = event_tx.send(app_event);` — ignore send errors (no receivers = events dropped safely)

    2. Define `pub async fn db_writer_task(mut event_rx: tokio::sync::broadcast::Receiver<AppEvent>, conn: tokio_rusqlite::Connection)`:
       - Loop: match on `event_rx.recv().await`
         - `Ok(event)` → call `conn.call(|conn| { store::insert_callback_event(conn, ...) }).await`
         - `Err(broadcast::error::RecvError::Lagged(n))` → log warning: `eprintln!("warning: db writer lagged, dropped {} events", n)` (Pitfall 4 from research)
         - `Err(broadcast::error::RecvError::Closed)` → break

    3. Define `pub async fn stdout_logger_task(mut event_rx: tokio::sync::broadcast::Receiver<AppEvent>, json_mode: bool)`:
       - Loop: match on `event_rx.recv().await`
         - `Ok(event)` →
           If `json_mode`: print `serde_json::to_string(&log_entry)` (per D-10 --json flag)
           Else: print structured one-liner per D-10: `"{timestamp} tier={tier} class={classification} ip={ip} ua=\"{ua_snippet}\""` where ua_snippet is first 60 chars of UA.
         - Handle `Lagged` and `Closed` same as db_writer.

    4. Update `src/lib.rs`: add `pub mod broker;`

    Note: The broker module uses `crate::types::{RawCallbackEvent, AppEvent, AgentClass}`, `crate::fingerprint::compute_session_id`, and `crate::store::insert_callback_event`. It does NOT use `http::HeaderMap` (that's the server's job).

    For unit tests: since broker_task is async and uses channels, write tests using `#[tokio::test]`. Add `tokio = { version = "1", features = ["macros", "rt-multi-thread"] }` to `[dev-dependencies]` in Cargo.toml if not already present. Test that sending a RawCallbackEvent through mpsc results in an AppEvent appearing on the broadcast receiver.
