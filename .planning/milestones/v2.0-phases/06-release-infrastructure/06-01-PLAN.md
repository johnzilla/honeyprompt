  <read_first>.github/workflows/ci.yml, .planning/phases/06-release-infrastructure/06-CONTEXT.md, .planning/research/STACK.md, .planning/research/PITFALLS.md, Cargo.toml</read_first>

  <action>
Create `.github/workflows/release.yml` implementing the cross-platform release pipeline.

**Workflow structure** (per D-01, D-02):
- Trigger: `push: tags: ['v[0-9]+.*']` (version tags only, no manual dispatch, no pre-release)
- Permissions: `contents: write` (needed to create releases and upload assets)

**Jobs:**

**Job 1: `create-release`**
- Runs on: `ubuntu-latest`
- Steps:
  1. `actions/checkout@{SHA}  # v4`
  2. `taiki-e/create-gh-release-action@{SHA}  # v1` with `token: ${{ secrets.GITHUB_TOKEN }}`
- This creates the GitHub Release entry from the tag

**Job 2: `upload-assets`**
- Needs: `create-release`
- Strategy matrix with `fail-fast: false` (per D-04, D-05, D-06):

```yaml
matrix:
  include:
    - target: x86_64-unknown-linux-musl
      os: ubuntu-latest
    - target: aarch64-unknown-linux-musl
      os: ubuntu-latest
    - target: x86_64-apple-darwin
      os: macos-latest
    - target: aarch64-apple-darwin
      os: macos-latest
```

- Runs on: `${{ matrix.os }}`
- Steps:
  1. `actions/checkout@{SHA}  # v4`
  2. `taiki-e/upload-rust-binary-action@{SHA}  # v1` with:
     - `bin: honeyprompt`
     - `target: ${{ matrix.target }}`
     - `tar: unix` (produces .tar.gz archives)
     - `token: ${{ secrets.GITHUB_TOKEN }}`
     - `checksum: sha256` (Claude's discretion: include checksums — standard security tool practice)

**Binary naming** (per D-03): The taiki-e action automatically names archives as `honeyprompt-{target}.tar.gz` which matches the full Rust target triple convention.

**Key details:**
- `taiki-e/upload-rust-binary-action` automatically installs `cross` when the target differs from the runner's native target (Linux musl targets on ubuntu). No manual cross installation needed.
- The `bundled` feature on rusqlite compiles SQLite from C source, which works with both musl (Linux) and native macOS SDK. Per Pitfall 4, macOS targets MUST run on macos-latest runners.
- Per D-05, Linux targets use musl for fully static binaries — maximum portability.
- Per D-07/D-08, use the same checkout/toolchain SHAs as ci.yml. The taiki-e actions use their own resolved SHAs from Task 1.
- Add `CARGO_TERM_COLOR: always` env var consistent with ci.yml.

**Do NOT include:** Windows targets (out of scope), crates.io publish step (deferred), cargo-dist (deferred per CONTEXT.md deferred section).
