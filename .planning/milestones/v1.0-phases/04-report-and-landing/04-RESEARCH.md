## Summary

Phase 4 delivers two distinct artifacts: a `report` CLI subcommand that reads from the existing SQLite event store and produces a Markdown disclosure document, and the honeyprompt.sh landing page as a committed project asset.

The report module is architecturally the simplest in the project — it is a pure synchronous function that opens the database, runs a set of parameterized queries, and formats the results as Markdown using Rust's standard string facilities. No new crates are required. The existing stack (rusqlite, serde_json, anyhow, clap) covers everything needed. The only risk is the Markdown injection pitfall: agent-supplied strings (UA, IP, extra_headers JSON) must be escaped before interpolation into the report body.

The landing page requires no new code whatsoever. It is a `honeyprompt.toml` config file committed to a `landing/` directory in the repository, with the generated `output/` files either committed or produced by a `make`/`just` target. The generator pipeline is already complete and reusable as-is. The discretionary decisions concern which callback URL to hard-code in the landing config and whether to commit the generated HTML to the repository.

**Primary recommendation:** Implement `src/report/mod.rs` as a pure synchronous module with two new store query functions (`query_report_sessions` and `query_report_summary`). The landing page is a config-plus-generation task with no code changes.

### Pattern 1: Synchronous Report Command (matches init/generate pattern)

**What:** The report command does not use Tokio. It opens the SQLite DB synchronously, queries all sessions, builds a ReportData struct, renders to Markdown, and writes to a file or stdout. This mirrors the `init` and `generate` subcommands exactly.

**When to use:** Any subcommand that only needs filesystem + SQLite and no network I/O.

**Example:**
```rust
// src/main.rs — follows the Commands::Generate pattern exactly
Commands::Report(args) => {
    let path = &args.path;
    let db_path = path.join(".honeyprompt").join("events.db");
    let conn = store::open_or_create_db(&db_path)?;
    let markdown = report::generate_report(&conn)?;
    if args.stdout {
        print!("{}", markdown);
    } else {
        let out_path = args.output.as_deref()
            .unwrap_or_else(|| std::path::Path::new("report.md"));
        std::fs::write(out_path, &markdown)?;
        println!("Report written to {}", out_path.display());
    }
    Ok(())
}
```

### Pattern 4: Markdown Rendering with String Escaping

**What:** Build Markdown using `format!` and `String::push_str`. All agent-supplied strings (UA, IP, classification, headers) MUST be passed through a `md_escape()` helper before interpolation. This is Pitfall 12.

**When to use:** Every field sourced from events table that contains agent-supplied data.

**Example:**
```rust
// src/report/mod.rs
fn md_escape(s: &str) -> String {
    // Escape Markdown metacharacters that could alter table structure or inject content
    // Minimum: | \n backtick  (pipe breaks table columns; newline breaks table rows)
    s.replace('|', "\\|")
     .replace('\n', " ")
     .replace('\r', "")
     .replace('`', "\\`")
}

fn render_evidence_row(session: &ReportSession) -> String {
    format!(
        "| {} | Tier {} | {} | {} | {} | {} | {} | {} |\n",
        md_escape(&session.session_id[..8]),  // abbreviated for readability
        session.tier,
        md_escape(&session.first_seen_at),
        md_escape(&session.remote_addr),
        md_escape(&session.user_agent),
        session.fire_count,
        md_escape(&session.classification),
        md_escape(&session.payload_id),
    )
}
```

### Pattern 5: Landing Page as Config + Generate

**What:** The landing page is produced by committing a `landing/honeyprompt.toml` to the repository and running `honeyprompt generate landing/` to produce `landing/output/`. No new code. The config sets `page_title`, `callback_base_url`, and `tiers`.

**When to use:** Landing page creation.

**Example config (landing/honeyprompt.toml):**
```toml
callback_base_url = "https://honeyprompt.sh"
bind_address = "0.0.0.0:8080"
tiers = [1, 2, 3]
page_title = "HoneyPrompt — AI Agent Detection"
theme = "default"
```

The `landing/output/` files are committed to the repository so the landing page can be deployed without running `generate`. The `.honeyprompt/events.db` is gitignored (it is runtime state, not a source artifact).

### Anti-Patterns to Avoid

- **Interpolating raw SQL into format!:** Never use `format!("SELECT ... WHERE ua = '{}'", user_agent)`. Use `params![]` as all prior modules do. The RUSTSEC-2025-0043 advisory is the canonical example of this failure mode.
- **Counting raw event rows:** Never `SELECT COUNT(*) FROM events` for the detection number. Use session-based counts as in the existing `count_detections()` function. Violates D-04.
- **Treating extra_headers as structured data without parsing:** The `extra_headers` column is a JSON blob (`{"classification":"KnownCrawler","headers":{...}}`). Extract classification by parsing via `serde_json::from_str` into a typed struct, not via LIKE pattern matching (the `count_detections` function uses a LIKE hack that is acceptable for counting but insufficient for extracting the provider name for the report).
- **Async for the report command:** Do not wrap report in a Tokio runtime. It is synchronous filesystem + SQLite work, matching `init` and `generate`.
- **Committing events.db:** The `.honeyprompt/events.db` is runtime state and must be gitignored. Only `landing/honeyprompt.toml` and `landing/output/` belong in source control for the landing page artifact.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Markdown table escaping | Custom parser | `md_escape()` helper (4 replacements) | The escaping surface is narrow: pipes, newlines, backticks — no need for a library |
| JSON parsing of extra_headers | Manual string extraction | `serde_json::from_str` | Already in Cargo.toml; handles nested structures and edge cases |
| Date range calculation | Custom timestamp arithmetic | SQL `MIN(first_seen_at)` / `MAX(last_seen_at)` | Timestamps stored as epoch seconds (see `chrono_now()` in store/mod.rs) — min/max in SQL is trivial |
| Session aggregation | Application-level grouping | SQL `GROUP BY session_id, tier` | The DB already has the right indexes; let SQLite do the work |

**Key insight:** The report module is deliberately simple. Its complexity budget goes entirely into the SQL queries and the escaping discipline. There is no Markdown library, no templating ceremony, and no async complexity.

## Common Pitfalls

### Pitfall 1: Markdown Injection from Agent-Supplied Strings (Pitfall 12 from PITFALLS.md)

**What goes wrong:** User-agent strings or IP addresses containing `|` characters break GFM table column alignment. Strings containing newlines break table rows. Backtick sequences produce unintended code spans. In worst case, an adversarial agent that has probed the tool sends a crafted UA string containing `\n| malicious | row |` which inserts a fake evidence row into the report.

**Why it happens:** String interpolation into Markdown without escaping — an easy oversight since Markdown does not have a formal quoting spec.

**How to avoid:** Apply `md_escape()` to every field sourced from the events table before interpolation. Test with a UA string containing `| pipe |` to confirm the output table is valid.

**Warning signs:** Report output with misaligned table columns, unexpected blank rows, or extra table rows not matching database row count.

### Pitfall 2: Classification Extraction from extra_headers JSON Blob

**What goes wrong:** The `classification` data is stored inside the `extra_headers` JSON blob as `{"classification":"KnownCrawler","headers":{...}}`. The existing `count_detections()` function extracts classification via a LIKE pattern (`extra_headers NOT LIKE '%"classification":"KnownCrawler"%'`). This pattern is fragile for display purposes — it cannot extract the provider name or handle `KnownAgent`.

**Why it happens:** The Phase 2 decision (from STATE.md) was to store classification in the extra_headers blob to avoid a schema migration. This was the right call then, but the report module needs to parse it properly.

**How to avoid:** Define a `ExtraHeaders` deserialization struct:
```rust
#[derive(Deserialize)]
struct ExtraHeaders {
    classification: String,   // "KnownCrawler" | "KnownAgent" | "Unknown"
    #[serde(default)]
    provider: Option<String>,
    headers: serde_json::Value,
}
```
Use `serde_json::from_str::<ExtraHeaders>(&extra_headers_str)` and handle parse errors gracefully (fall back to `"Unknown"` on malformed JSON — the field was never guaranteed to always be parseable).

**Warning signs:** Report showing all sessions classified as "Unknown" when the DB contains KnownCrawler/KnownAgent events, or panic on serde parse error.

### Pitfall 3: Session Deduplication in Report Evidence Table

**What goes wrong:** The evidence table should show one row per (session_id, tier) pair (per D-04), but a naive `SELECT * FROM events` query returns one row per nonce fire. A session that fired Tier 1 and Tier 2 callbacks should appear as two rows in the table (different tiers, same session), but a session that replayed the same nonce twice (`is_replay = 1`) should appear as one row.

**Why it happens:** The events table has one row per unique nonce, and fire_count tracks replay. The GROUP BY query must group by session_id + tier, taking the first-seen values and summing fire counts.

**How to avoid:** Use:
```sql
SELECT session_id, tier, payload_id, embedding_loc,
       MIN(first_seen_at) as first_seen_at,
       MAX(last_seen_at) as last_seen_at,
       SUM(fire_count) as total_fires,
       MAX(remote_addr) as remote_addr,
       MAX(user_agent) as user_agent,
       MAX(extra_headers) as extra_headers
FROM events
GROUP BY session_id, tier
ORDER BY MIN(first_seen_at) DESC
```
Note: `MAX(remote_addr)` is a pragmatic choice since within a session the remote_addr should be consistent. Document the assumption.

**Warning signs:** Evidence table row count exceeds number of unique (session_id, tier) pairs.

### Pitfall 4: Landing Page events.db in Git

**What goes wrong:** Running `honeyprompt generate landing/` creates `landing/.honeyprompt/events.db`. If this file is committed, the repository accumulates binary data with every deployment. Worse, it may contain real callback events from the live site.

**How to avoid:** Ensure `.honeyprompt/` is in `.gitignore` at both the root and landing directory level. The `landing/output/` HTML files are static and safe to commit; the SQLite DB is runtime state.

**Warning signs:** `git status` showing `landing/.honeyprompt/events.db` as untracked after running generate.

### Pitfall 5: Report date range with epoch-second timestamps

**What goes wrong:** Timestamps in the events table are stored as Unix epoch seconds (plain integers as strings, per the `chrono_now()` implementation in store/mod.rs). The report executive summary needs a human-readable date range. Converting epoch seconds to ISO-8601 requires either the `chrono` crate or manual formatting.

**Why it happens:** Phase 1 decision (STATE.md: "chrono_now() uses std::time::SystemTime for timestamps — avoids adding time crate in Phase 1"). The timestamps are stored as `"{secs}"` (stringified integer).

**How to avoid:** Add `chrono` as a dependency in this phase for timestamp formatting in the report. The alternative — manual division/modulo to format dates — is error-prone. Chrono is the standard Rust date library and will not cause any compatibility issues with the existing stack.

**Warning signs:** Report showing "Timestamp: 1743289056" instead of "2026-03-29T12:17:36Z" in the date range.

## Open Questions

1. **Whether to add the `chrono` crate for timestamp formatting**
   - What we know: Timestamps stored as Unix epoch seconds (string representation of u64). The `chrono` crate is the standard for Rust date formatting, widely used, no known conflicts with the stack.
   - What's unclear: Whether `chrono` is already an indirect dependency (check `cargo tree`). If it is, just add it to `[dependencies]`. If not, the binary size increase is minimal (~200KB).
   - Recommendation: Add `chrono = { version = "0.4", features = ["serde"] }` in the plan. The alternative (manual epoch formatting) is error-prone and not worth the saved dependency.

2. **Whether `landing/output/` should be committed to the repository**
   - What we know: The generator produces static HTML files. Committing them means the landing page is always deployable from a `git clone` without running `generate`. Not committing means the CI/deployment step must run `generate`.
   - What's unclear: How honeyprompt.sh will be deployed (static hosting, VPS, etc.).
   - Recommendation: Commit `landing/output/` as static files. The purpose of LAND-01 is to prove the tool works on itself — a committed output that can be inspected directly makes that proof tangible without requiring a deployment step. The planner should make the final call based on deployment context.

3. **Provider name in classification display**
   - What we know: `AgentClass::KnownCrawler { provider }` carries a provider string, but this is stored in `extra_headers` JSON. The exact JSON key for provider is not confirmed from the existing code — the Phase 2 fingerprint module stores it, but the schema of the stored JSON is implied by the test data which only shows `{"classification":"KnownAgent","headers":{...}}` without an explicit `provider` field.
   - What's unclear: Does the fingerprint module store the provider name in extra_headers? Or only the classification tier?
   - Recommendation: Before writing `query_report_sessions`, read `src/fingerprint/mod.rs` to confirm the exact JSON structure. The plan should include this as a prerequisite step.
