## Overview

Extends the graduated proof model from Tiers 1–3 to Tiers 4 (Capability Introspection) and 5 (Multi-step Compliance Chain), giving defenders deeper verifiable evidence of AI agent compliance — still with no secrets transmitted, and fully backward-compatible with v1.0–v4.0 deployments.

**Milestone Goal:** Backward-compatible, no-secrets extension of the graduated proof model. `/cb/v1/` behavior frozen. SQLite migration additive. T4 list items agent-chosen from a safe menu. T5 proofs are arithmetic of page-visible values.

**Cross-cutting constraints (enforced across every v5.0 phase):**
- `/cb/v1/{nonce}` route behavior frozen — T1–T3 callbacks unchanged
- SQLite migrations additive only — existing T1–T3 rows remain readable without transformation
- No secrets ever leave the agent — T4 lists are agent-chosen from a safe menu; T5 proofs are arithmetic of page-visible values
- Five embedding locations stay fixed (HTML comment, meta, hidden span, JSON-LD, prose)

## Milestone Summary

**Decimal Phases:** None — all v5.0 phases were planned phases (13, 14, 15), no gap-closure inserts.

**Key Decisions (added to PROJECT.md):**

- **3-phase split** (Backend → Surfacing → Validation+Docs) chosen for smallest blast radius with demoable boundaries at each phase
- **Backward-compat concentrated in Phase 13** since that's where the risk lives; Phase 14 and 15 are additive on top
- **`verification_seed` derived from nonce** (no seed storage) — `u32::from_str_radix(&nonce[0..8], 16)` avoids a schema column and keeps replay semantics uniform
- **T5 formula shape** uniform across templates: `((seed + a) * b) % mod` as zero-padded 3-digit decimal; constants vary per template, instructions vary in wording
- **T4 sanitization** regex `^[a-z0-9_,.\-]{1,256}$` after base64 decode + lowercase normalization; malformed → 204 with nothing stored
- **Always-show chrome policy** (D-14-12) — TUI stats header and report exec summary always render T4/T5 rows with count=0 so defenders see the full 5-tier model regardless of DB contents
- **Self-identifying T5 seed JSON-LD blocks** — each carries `verification_seed` and `nonce` so multiple T5 blocks on one page disambiguate; server uses URL-path nonce as authoritative
- **Symmetric scorecard extension** (D-15-01) — same three verdict values (NO_COMPLIANT, PARTIALLY_COMPLIANT, FULLY_COMPLIANT); FULLY just now requires all 5 tiers
- **Fixed-size `[bool; 5]`/`[u32; 5]` over Vec** (D-15-05) — simplest change, matches prior pattern, v5.0 is terminal for tier expansion
- **README "inline parenthetical" example style** (D-15-08) — preserves existing 5-bullet density, minimal visual noise

**Issues Resolved:**

- Phase 13 dot-zero integration bug in T5 seed distribution surfaced and fixed by Phase 14-01 (T5Formula propagation through the event pipeline was never exercised end-to-end until TUI rendering demanded it)
- Pre-existing setup wizard bugs (T4/T5 menu missing 5-tier option, DB parent dir auto-create, attach-mode SELECT) folded in as gap fixes during Phase 14 execution
- TUI contrast (DarkGray → Gray for readable contrast on dark terminals) folded in during Phase 14 execution
- Phase 14 TUI `attach` mode `t5_proof_valid` was read as `bool` instead of `Option<i64>` — fixed and surfaced DB errors to status line (post-commit fix)
- README T5 worked-example constants drafted as `(42, 7, 1000)` in CONTEXT.md didn't match any shipped template; planner corrected to `(42, 17, 1000)` from `t5-semantic-prose` during Phase 15 planning (authorized by CONTEXT.md `<specifics>`)
